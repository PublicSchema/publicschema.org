# Build-scripts cutover plan

The PublicSchema build pipeline ships four CLI scripts that read the
bespoke schema YAML directly. After the LinkML migration lands, the
canonical source of truth becomes `dist/linkml/`, regenerated by
`build/migrate_to_linkml.py`. This note records what each script does
today, what changed in this PR, and what is left for the cutover.

## `ps-validate` (`build/validate.py`) â€” adapted

Today the script runs JSON Schema validation, referential integrity,
multilingual completeness, age_applicability cross-checks against
bibliography citations, and property_groups completeness. The full rule
set has no direct equivalent in stock LinkML tooling: `linkml-validate`
validates *data instances*, and `linkml-lint --validate` only covers
the metamodel.

Change: kept the bespoke rule set as the default, added a
`--source linkml` flag that delegates to `linkml-lint --validate
--ignore-warnings` on `dist/linkml/publicschema.yaml`. After cutover
each rule must be re-implemented against the LinkML field shape; a
concrete plan is recorded in the module docstring.

## `ps-lint` (`build/lint.py`) â€” adapted

Custom content linter (jargon, definition quality, em-dash, maturity
gates, external_equivalents URI checks). Stock `linkml-lint` does not
replace any of these rules.

Change: extracted a `_lint_loaded` core that runs against pre-loaded
concept/property/vocabulary dicts. Added a LinkML reader
(`load_linkml_as_bespoke`) that projects each class/slot/enum back to
the bespoke shape by reading `title` (en label), `description` (en
definition), and the `annotations.label_*` / `description_*` /
`external_alignments_json` / `convergence_json` / `property_groups_json`
/ `sync_json` annotations. `--source linkml` reuses the same rule set
unchanged. Verified locally: both readers produce comparable issue
counts (82 bespoke vs 83 linkml â€” drift is one rule that splits an
acronym list differently after JSON round-trip).

## `ps-check-translations` (`build/check_translations.py`) â€” adapted

The schema check requires every candidate/normative entity to carry FR
and ES `definition` (and `label`, when present). The UI/docs/prose
checks are independent of schema source.

Change: added `check_schema_linkml` plus `--source linkml`. The reader
maps LinkML `title` â†’ `label.en`, `description` â†’ `definition.en`,
`annotations.label_fr/es` â†’ `label.fr/es`, `annotations.description_fr/
es` â†’ `definition.fr/es`, then runs the same `_check_definition` /
`_check_label` helpers. Both paths report zero schema errors today.

## `ps-sync` (`build/sync_standards.py`) â€” kept (with TODO)

Pulls vocabulary values from authoritative external sources (FHIR, ISO,
SIL, etc.) and **writes back to `schema/vocabularies/**/*.yaml`**.
Because writes target the source-of-truth tree, the script intentionally
stays on the bespoke tree until cutover; any LinkML write would be
overwritten by the next migrator run. The module docstring records the
two cutover options (rewrite to read/write LinkML `annotations.sync_json`
and `permissible_values`, vs. keep writing bespoke during a deprecation
window) and the helper that needs the new `--source linkml` mode
(`find_syncable_vocabularies`).
